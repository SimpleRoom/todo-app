# Feature Specification: 任务清单应用

**Feature Branch**: `001-task-list`

**Created**: 2026-06-29

**Status**: Draft

**Input**: User description: "实现一个任务清单应用，功能包括：新增任务（标题和描述）、删除任务、标记完成/取消完成、状态筛选（全部、未完成、已完成）、localStorage 持久化"

## Clarifications

### Session 2026-06-29

- Q: 多标签页并发场景下的行为？ → A: 多标签页自动同步（通过浏览器 storage 事件实现跨标签页实时同步，无数据丢失）
- Q: 是否需要支持编辑已有任务的标题与描述？ → A: 支持编辑（保留原 createdAt 与 completed 状态，仅更新标题与描述）
- Q: 未完成任务列表内部应如何排序？ → A: 维持创建时间倒序不变，已完成任务保持在原创建时间位置，不单独置底
- Q: 是否需要批量删除已完成任务？ → A: 支持批量删除（多选 + 全选 + 批量删除按钮 + 二次确认，需提示选中数量）
- Q: 数据导入导出应采用什么形式？ → A: JSON 文件下载/上传，通过设置入口触发，导入时合并到当前列表（重复 id 跳过），含格式校验

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 新增与查看任务 (Priority: P1)

用户能在页面顶部的新增区域输入任务标题（必填）与任务描述（可选），点击"添加"按钮后任务立即出现在下方列表中。刷新浏览器后，已添加的任务依然存在。

**Why this priority**: 没有新增与查看，就没有任务清单可言。这是整个应用的最小可用切片，独立可用即可作为 MVP 交付，并验证 localStorage 持久化链路。

**Independent Test**: 在新增区域填写标题与描述并添加 ≥1 条任务后，刷新浏览器，任务依然显示在列表中。即可视为本故事独立可用。

**Acceptance Scenarios**:

1. **Given** 新增区域标题输入框为空，**When** 用户点击"添加"按钮，**Then** 添加操作被阻止并显示"请输入任务标题"的中文提示，列表不变。
2. **Given** 用户已在标题输入框填写"买牛奶"、描述框填写"下班顺路"，**When** 用户点击"添加"按钮，**Then** 列表顶部出现一条新任务，标题为"买牛奶"、描述为"下班顺路"，状态为未完成，创建时间为当前时间。
3. **Given** 用户已添加 3 条任务，**When** 用户刷新浏览器，**Then** 列表依然显示这 3 条任务，顺序、标题、描述、状态均与刷新前一致。
4. **Given** 用户仅填写标题未填写描述，**When** 用户点击"添加"按钮，**Then** 任务被成功创建，描述为空且界面正常显示。

---

### User Story 2 - 完成状态切换 (Priority: P2)

用户能通过任务条目上的复选框或操作按钮，在"未完成"与"已完成"之间切换任意任务的状态，状态变化立即反映在界面上（如标题加删除线、复选框打勾），并立即持久化。

**Why this priority**: 完成状态是任务清单的核心语义。在能看到任务（US1）之后，标记完成是用户最高频的下一步操作。

**Independent Test**: 添加至少 1 条任务后，点击其完成切换控件，状态在"未完成 ↔ 已完成"之间切换，刷新浏览器后状态保持。

**Acceptance Scenarios**:

1. **Given** 列表中存在 1 条未完成任务"写周报"，**When** 用户点击该任务的完成切换控件，**Then** 该任务状态变为已完成，标题显示删除线，复选框呈现选中状态。
2. **Given** 列表中存在 1 条已完成任务"写周报"，**When** 用户再次点击该任务的完成切换控件，**Then** 该任务状态变为未完成，删除线消失，复选框恢复未选中状态。
3. **Given** 用户刚将任务切换为已完成，**When** 用户刷新浏览器，**Then** 该任务依然显示为已完成状态。

---

### User Story 3 - 删除任务 (Priority: P3)

用户能删除任意任务。删除前要求二次确认，确认后才真正删除；删除后任务从列表中消失并立即持久化。

**Why this priority**: 删除是不可逆操作（即使本应用提供撤销，列表层面也是不可逆显示），需要谨慎。在能用、能完成之后，清理已完成或误建的任务是自然的下一步。

**Independent Test**: 添加至少 1 条任务后，点击其删除按钮并在确认弹窗中点击"确认"，任务从列表消失，刷新浏览器后依然消失。

**Acceptance Scenarios**:

1. **Given** 列表中存在任务"买牛奶"，**When** 用户点击该任务的"删除"按钮，**Then** 弹出中文确认弹窗，提示"确定要删除该任务吗？"，包含"确认"与"取消"两个按钮，列表暂未变化。
2. **Given** 删除确认弹窗已弹出，**When** 用户点击"取消"，**Then** 弹窗关闭，列表保持不变，任务"买牛奶"依然存在。
3. **Given** 删除确认弹窗已弹出，**When** 用户点击"确认"，**Then** 弹窗关闭，任务"买牛奶"从列表中消失，刷新浏览器后该任务依然不存在。

---

### User Story 4 - 状态筛选 (Priority: P4)

用户能通过列表上方的筛选器在"全部"、"未完成"、"已完成"三种视图之间切换，列表只显示符合当前筛选条件的任务。筛选状态本身在刷新后保持。

**Why this priority**: 当任务数量增多后，筛选是必要的导航能力。但应用在少量任务时依然可用，故为 P4 渐进增强。

**Independent Test**: 添加若干任务并标记其中部分为已完成后，分别切换三个筛选标签，列表内容随筛选条件变化，刷新后筛选状态保持。

**Acceptance Scenarios**:

1. **Given** 列表中存在 2 条未完成任务与 1 条已完成任务，当前筛选为"全部"，**When** 用户点击"未完成"筛选标签，**Then** 列表只显示 2 条未完成任务，已完成任务被隐藏，"未完成"标签呈现选中态。
2. **Given** 同上初始状态，**When** 用户点击"已完成"筛选标签，**Then** 列表只显示 1 条已完成任务，未完成任务被隐藏，"已完成"标签呈现选中态。
3. **Given** 当前筛选为"未完成"，**When** 用户刷新浏览器，**Then** 刷新后筛选状态依然为"未完成"，列表依然只显示未完成任务。
4. **Given** 当前筛选为"未完成"，列表中所有未完成任务被依次标记为已完成，**When** 列表更新后，**Then** 列表显示为空，并出现"暂无未完成任务"的中文空状态提示。

---

### User Story 5 - 编辑任务 (Priority: P3)

用户能编辑已有任务的标题与描述，进入编辑态后修改内容并保存，保存后任务保留原始 `createdAt` 与 `completed` 状态，仅标题与描述更新。

**Why this priority**: 编辑与删除（US3）同属"管理已有任务"的操作层级，可与之并行实现。在能用（US1）、能完成（US2）之后，修正错别字或补充描述是高频需求；不提供编辑会迫使用户删除重建，丢失 `createdAt`，违反宪章 IV"数据可靠"。

**Independent Test**: 添加 1 条任务后，进入其编辑态、修改标题与描述并保存，列表显示更新后的内容且 `createdAt` 与完成状态保持不变，刷新浏览器后编辑结果依然存在。

**Acceptance Scenarios**:

1. **Given** 列表中存在任务"买牛奶"（描述为"下班顺路"），**When** 用户点击该任务的"编辑"按钮，**Then** 该任务进入编辑态，标题与描述变为可编辑输入框，预填当前内容，并显示"保存"与"取消"按钮。
2. **Given** 任务处于编辑态，用户已将标题修改为"买酸奶"、描述修改为"早上路过超市"，**When** 用户点击"保存"，**Then** 编辑态退出，列表中该任务标题显示为"买酸奶"、描述为"早上路过超市"，`createdAt` 与 `completed` 状态保持不变，并立即持久化。
3. **Given** 任务处于编辑态，用户已修改标题，**When** 用户点击"取消"，**Then** 编辑态退出，任务标题与描述恢复为编辑前的内容，无任何持久化发生。
4. **Given** 任务处于编辑态，用户清空标题输入框，**When** 用户点击"保存"，**Then** 保存被阻止并显示"请输入任务标题"的中文提示，编辑态保持。
5. **Given** 任务处于编辑态，用户已修改标题并保存，**When** 用户刷新浏览器，**Then** 刷新后任务依然显示编辑后的标题与描述。

---

### User Story 6 - 批量删除任务 (Priority: P4)

用户能通过多选 + 批量删除一次性清理多条任务，避免对大量已完成任务逐条删除的繁琐操作。批量删除仍需二次确认。

**Why this priority**: 在能用（US1）、能完成（US2）、能单条删除（US3）之后，当任务累积至数十条时，逐条删除效率低。批量操作是渐进增强，依赖前序故事但独立可用。与宪章 I"简洁优先"的张力通过将多选作为可选模式（默认隐藏）化解。

**Independent Test**: 添加 ≥3 条任务后，进入多选模式选中其中 2 条，点击"批量删除"并在确认弹窗点击"确认"，被选中的 2 条从列表消失，未选中的任务保持不变，刷新浏览器后依然消失。

**Acceptance Scenarios**:

1. **Given** 列表中存在 4 条任务，当前为普通视图，**When** 用户点击"批量操作"按钮，**Then** 列表进入多选模式：每条任务前出现复选框，顶部出现"全选"、"删除所选"、"退出多选"按钮，新增区域隐藏。
2. **Given** 列表处于多选模式，存在 4 条任务均未选中，**When** 用户点击"全选"，**Then** 4 条任务全部被选中，"全选"按钮文案变为"取消全选"。
3. **Given** 列表处于多选模式，用户已选中 2 条任务，**When** 用户点击"删除所选"，**Then** 弹出中文确认弹窗，提示"确定要删除选中的 2 条任务吗？"，包含"确认"与"取消"按钮，列表暂未变化。
4. **Given** 批量删除确认弹窗已弹出（选中 2 条），**When** 用户点击"确认"，**Then** 弹窗关闭，被选中的 2 条任务从列表消失，未选中的任务保持不变，多选模式保持，已选数量重置为 0，立即持久化。
5. **Given** 批量删除确认弹窗已弹出，**When** 用户点击"取消"，**Then** 弹窗关闭，列表保持不变，选中状态保持。
6. **Given** 列表处于多选模式，**When** 用户点击"退出多选"，**Then** 列表退出多选模式，复选框与批量工具栏消失，新增区域恢复显示，所有选中状态清空。
7. **Given** 列表处于多选模式且无任何任务被选中，**When** 用户点击"删除所选"，**Then** 操作被阻止并显示"请先选择要删除的任务"的中文提示，不弹出确认弹窗。

### User Story 7 - 数据导入导出 (Priority: P4)

用户能通过设置入口将全部任务导出为 JSON 文件备份，或将 JSON 文件导入合并到当前任务列表，以实现数据备份与跨浏览器/设备迁移。

**Why this priority**: 宪章 IV"数据可靠"明确要求"提供数据导出（JSON）与导入能力，用户不被锁定"。这是数据可靠性的兜底能力，但不阻塞核心任务流程，故为 P4 渐进增强。

**Independent Test**: 添加 ≥2 条任务后，通过设置入口导出 JSON 文件，文件包含全部任务字段；清空 localStorage 后重新打开应用，通过设置入口导入该 JSON 文件，任务列表恢复。

**Acceptance Scenarios**:

1. **Given** 列表中存在 3 条任务，**When** 用户点击设置入口的"导出数据"按钮，**Then** 浏览器触发文件下载，文件名为 `todo-backup-YYYYMMDD-HHMMSS.json`（中文时间格式对应的数字形式），文件内容为 JSON 数组，每个元素包含 `id`、`title`、`description`、`completed`、`createdAt` 五个字段。
2. **Given** 当前列表为空，**When** 用户点击设置入口的"导入数据"按钮并选择一个合法的 JSON 备份文件（含 2 条任务），**Then** 列表合并显示 2 条任务，原有空列表变为 2 条，立即持久化。
3. **Given** 列表中已存在任务 A（`id: "a1"`），**When** 用户导入含任务 A（`id: "a1"`）与任务 B（`id: "b2"`）的 JSON 文件，**Then** 任务 A 保持当前本地内容不变（重复 id 跳过），任务 B 被添加到列表，立即持久化。
4. **Given** 用户选择一个非 JSON 文件或格式损坏的文件，**When** 触发导入，**Then** 导入被阻止并显示"文件格式无效，请选择有效的备份文件"的中文提示，现有列表不变。
5. **Given** 用户选择一个 JSON 数组但元素缺少必填字段（如 `title` 缺失），**When** 触发导入，**Then** 缺字段的记录被跳过并在控制台告警，合法记录正常合并，界面显示"成功导入 N 条，跳过 M 条无效记录"。

### Edge Cases

- **localStorage 禁用或不可写**（如 Safari 隐私模式）：应用 MUST 在控制台输出警告并降级为内存态，界面仍可正常使用，刷新后数据丢失，并提示用户"当前环境无法持久化数据"。
- **localStorage 配额超限**：当写入失败时，新增/编辑/删除操作 MUST 不静默失败，MUST 向用户提示"存储空间已满，请删除部分任务后重试"。
- **数据格式异常**：当 localStorage 中既有数据格式损坏或字段缺失（如老版本数据缺少字段），应用 MUST 容错加载，能解析的任务正常显示，无法解析的记录跳过并在控制台告警，不阻塞界面。
- **标题超长**：当用户输入超过 100 字符的标题时，应用 MUST 限制输入或截断并提示"标题最多 100 字符"。
- **描述超长**：当用户输入超过 500 字符的描述时，应用 MUST 限制输入或截断并提示"描述最多 500 字符"。
- **空列表状态**：当列表无任务时，MUST 显示中文空状态提示"还没有任务，添加第一条吧"，而非空白页面。
- **筛选下空列表**：见 US4 验收场景 4。
- **重复标题**：应用 MUST 允许重复标题，不做唯一性约束。
- **多标签页并发同步**：当用户在多个标签页打开应用时，一个标签页的增删改 MUST 通过 storage 事件同步到其他标签页；若两个标签页几乎同时操作不同任务，两条变更 MUST 都被保留；若同时操作同一任务（罕见），后写入者覆盖先写入者，但 MUST 不影响其他任务。
- **编辑态与并发变更冲突**：当任务处于编辑态时，若另一标签页已删除该任务，用户点击"保存"时 MUST 提示"该任务已被删除"并退出编辑态，不写入新数据；若另一标签页修改了该任务，本标签页保存时 MUST 以本标签页的编辑结果为准（最后写入者覆盖），不影响其他任务。
- **多选模式与并发删除冲突**：在多选模式下，若另一标签页已删除部分当前选中的任务，本标签页点击"删除所选"时 MUST 仅删除当前仍存在的选中任务，确认弹窗中的数量 MUST 反映实际可删除的任务数，MUST 不报错。
- **多选模式与筛选切换交互**：在多选模式下切换筛选标签时，已选中的任务若被筛选隐藏 MUST 仍保持选中状态（仅在"全部"视图下可见时显示选中态），切换回包含该任务的视图时复选框依然勾选。
- **导入文件字段超长**：导入的 JSON 记录中 `title` 超过 100 字符或 `description` 超过 500 字符时，MUST 截断至上限并在控制台告警，不阻塞该记录导入。
- **导入文件 createdAt 异常**：导入记录的 `createdAt` 为非法值（如负数、非数字、非日期字符串）时，MUST 用导入时刻的时间戳替换并在控制台告警，不阻塞该记录导入。
- **导出空列表**：当前列表为空时点击"导出数据"，MUST 仍触发下载，文件内容为空 JSON 数组 `[]`，不报错。
- **导入触发跨标签页同步**：导入操作 MUST 通过 localStorage 写入，因此其他打开的标签页 MUST 自动同步到导入后的完整列表（遵循 FR-012）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个新增区域，包含标题输入框（必填）、描述输入框（可选）和"添加"按钮。
- **FR-002**: 系统 MUST 在用户点击"添加"时校验标题非空（去除首尾空白后），为空则阻止添加并显示中文提示"请输入任务标题"。
- **FR-003**: 系统 MUST 为每条新任务自动生成唯一标识、记录创建时间（毫秒精度时间戳，展示精度到分钟），状态默认为"未完成"。
- **FR-004**: 系统 MUST 将任务列表以创建时间倒序展示（最新任务在顶部）。
- **FR-005**: 系统 MUST 在每条任务上提供完成状态切换控件，点击即在"未完成"与"已完成"之间切换，界面立即更新。
- **FR-006**: 系统 MUST 在每条任务上提供"删除"按钮，点击后弹出二次确认弹窗，仅当用户点击"确认"才执行删除。
- **FR-007**: 系统 MUST 在列表上方提供"全部"、"未完成"、"已完成"三个筛选标签，点击后列表立即按筛选条件刷新。
- **FR-008**: 系统 MUST 在当前选中的筛选标签上呈现清晰的选中态（如颜色或下划线），未选中的标签呈现默认态。
- **FR-009**: 系统 MUST 在每次新增、状态切换、删除操作后立即将最新任务列表持久化到浏览器 localStorage。
- **FR-010**: 系统 MUST 在应用启动时从 localStorage 读取任务列表并渲染；若读取失败或数据格式异常，MUST 容错降级而不阻塞界面。
- **FR-011**: 系统 MUST 将当前筛选状态也持久化到 localStorage，刷新后保持。
- **FR-012**: 系统 MUST 通过浏览器 storage 事件实现跨标签页自动同步——任一标签页的任务新增、状态切换、删除、筛选切换 MUST 在其他同时打开的标签页中自动反映，无需手动刷新，且 MUST 不产生数据丢失或覆盖。
- **FR-013**: 系统 MUST 在列表为空时显示中文空状态提示"还没有任务，添加第一条吧"；在筛选下无匹配任务时显示对应的中文空状态（如"暂无未完成任务"）。
- **FR-014**: 系统 MUST 限制标题最多 100 字符、描述最多 500 字符，超出时阻止输入或截断并给出中文提示。
- **FR-015**: 所有用户可见文本 MUST 为简体中文，包括按钮、标签、提示、空状态、错误信息、日期格式。
- **FR-016**: 系统 MUST 使用中文日期时间格式显示创建时间（YYYY年MM月DD日 HH:MM）。
- **FR-017**: 系统 MUST 在每条任务上提供"编辑"入口，点击后该任务进入编辑态，标题与描述变为可编辑输入框，预填当前内容，并提供"保存"与"取消"按钮。
- **FR-018**: 系统 MUST 在编辑态"保存"时校验标题非空（去首尾空白后），为空则阻止保存并显示"请输入任务标题"。
- **FR-019**: 系统 MUST 在编辑保存成功后保留原任务的 `id`、`createdAt`、`completed` 不变，仅更新 `title` 与 `description`，并立即持久化。
- **FR-020**: 系统 MUST 在编辑态"取消"时丢弃所有修改，恢复编辑前内容，不触发任何持久化。
- **FR-021**: 系统 MUST 在编辑态下对标题与描述施加与新增相同的字符上限（100 / 500）。
- **FR-022**: 系统 MUST 提供"批量操作"入口按钮，点击后列表进入多选模式：每条任务前出现选择复选框，顶部出现"全选/取消全选"、"删除所选"、"退出多选"按钮，新增区域隐藏。
- **FR-023**: 系统 MUST 在多选模式下，用户可通过单条任务的复选框切换其选中态；"全选"按钮 MUST 在所有任务已选中时文案变为"取消全选"并取消全部选中。
- **FR-024**: 系统 MUST 在多选模式下点击"删除所选"时校验至少选中 1 条任务，否则阻止操作并显示"请先选择要删除的任务"。
- **FR-025**: 系统 MUST 在批量删除时弹出二次确认弹窗，提示明确包含选中数量（如"确定要删除选中的 N 条任务吗？"），仅当用户点击"确认"才执行删除。
- **FR-026**: 系统 MUST 在批量删除确认后一次性删除所有选中任务，未选中任务保持不变，并立即持久化。
- **FR-027**: 系统 MUST 在多选模式下点击"退出多选"时退出多选模式、清空所有选中状态、恢复新增区域显示。
- **FR-028**: 系统 MUST 在筛选状态下也支持多选模式；"全选"只作用于当前筛选下可见的任务。
- **FR-029**: 系统 MUST 提供设置入口（如齿轮图标），包含"导出数据"与"导入数据"两个按钮。
- **FR-030**: 系统 MUST 在点击"导出数据"时将当前全部任务序列化为 JSON 数组并触发文件下载，文件名格式为 `todo-backup-YYYYMMDD-HHMMSS.json`，每个元素包含 `id`、`title`、`description`、`completed`、`createdAt` 五个字段。
- **FR-031**: 系统 MUST 在点击"导入数据"时弹出文件选择器，仅接受 `.json` 文件。
- **FR-032**: 系统 MUST 在导入时将 JSON 文件中的任务合并到当前列表：与本地现有任务 `id` 重复的记录跳过（保留本地版本），新 `id` 的记录添加到列表，立即持久化。
- **FR-033**: 系统 MUST 对导入文件做格式校验：非 JSON 文件、非数组结构、元素缺必填字段（`id`/`title`/`completed`/`createdAt`）的记录 MUST 被跳过；合法记录正常合并。
- **FR-034**: 系统 MUST 在导入完成后显示中文结果提示，包含"成功导入 N 条，跳过 M 条无效记录"的具体数量。
- **FR-035**: 系统 MUST 在导入文件完全无法解析（非 JSON 或非数组）时阻止导入并显示"文件格式无效，请选择有效的备份文件"，现有列表不变。

### Key Entities *(include if feature involves data)*

- **Task（任务）**: 任务清单的核心实体。
  - `id`: 字符串，唯一标识，新增时自动生成
  - `title`: 字符串，任务标题，非空，≤100 字符
  - `description`: 字符串，任务描述，可空，≤500 字符
  - `completed`: 布尔值，是否已完成，新增时为 `false`
  - `createdAt`: 字符串/数字，创建时间戳，精确到秒
- **FilterState（筛选状态）**: 枚举值，表示当前列表筛选条件。
  - 取值范围：`"all"`（全部）、`"active"`（未完成）、`"completed"`（已完成）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在 5 秒内完成"打开应用 → 输入标题 → 点击添加 → 看到任务出现在列表"的完整新增流程。
- **SC-002**: 用户刷新浏览器 10 次后，所有任务的标题、描述、完成状态、创建时间、顺序均与首次添加时完全一致。
- **SC-003**: 用户能在 3 秒内完成"点击复选框 → 看到任务状态切换并加删除线"的完整切换流程。
- **SC-004**: 用户能在 5 秒内完成"点击删除 → 在确认弹窗点击确认 → 看到任务从列表消失"的完整删除流程，且 100% 的误点击可通过"取消"按钮避免数据丢失。
- **SC-005**: 用户在含 20 条混合状态任务的列表中，能在 2 秒内通过筛选标签切换到"已完成"视图并仅看到已完成任务。
- **SC-006**: 100% 的用户可见文本为自然简体中文，无英文残留、无未翻译占位符、无生硬机翻表述。
- **SC-007**: 在浏览器隐私模式或 localStorage 禁用场景下，应用 MUST 仍可正常新增、切换、删除任务（仅刷新后不持久化），不出现白屏或未捕获异常。
- **SC-008**: 应用首屏渲染（从打开到可交互）在主流现代浏览器（Chrome、Edge、Safari、Firefox 最新版）下均不超过 1 秒。
- **SC-009**: 用户在标签页 A 新增任务后，切换到已打开的标签页 B 时，B MUST 无需手动刷新即可看到该任务，延迟不超过 1 秒。

## Assumptions

- 目标用户为单机单用户使用场景，不涉及多端同步、多用户协作、账号体系。
- 单个用户的任务总数预期在百条量级以内，不超过 localStorage 典型配额（5MB）。
- 用户使用现代浏览器（最近 2 年版本的 Chrome、Edge、Safari、Firefox），不支持 IE。
- 应用以单页应用（SPA）形式部署，可直接通过 `index.html` 打开或静态托管，无需服务端。
- 简体中文为唯一默认语言，英文界面不在本次范围内。
- 任务排序仅按创建时间倒序，不支持拖拽排序、优先级排序（YAGNI，遵循宪章"简洁优先"）。
- 不引入分类、标签、提醒、子任务等高级能力，保持 MVP 简洁（遵循宪章"简洁优先"与"渐进增强"）。
- 删除后不提供独立的撤销窗口（仅二次确认），以保持界面简洁；如未来需要可另行迭代。
- 日期时间显示精度到分钟已足够用户区分，无需显示秒。
